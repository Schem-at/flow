# Flow

Visual, node-based editor for building and running Minecraft schematic scripts.

Bun + Turborepo monorepo. Self-contained: the sandboxed JS execution engine
(`@flow/synthase`) and the isomorphic core (`@flow/core`) live in `packages/`.

## Layout

```
client/            React + Vite editor UI
server/            Hono + Bun backend
shared/            Shared types/utilities
packages/core      @flow/core   — isomorphic schematic execution engine
packages/synthase  @flow/synthase — sandboxed JS execution engine
```

## Develop

```bash
bun install
bun run dev          # turbo dev --filter=client
bun run build        # turbo build (all workspaces)
bun run test         # turbo test
```

## Use as a submodule of schemati

This repo is consumed by [`schemati`](https://github.com/Schem-at) as a git submodule
mounted at `schemati/flow`, with two docker compose services:

- **`flow`** — the Vite client on port 5176 (`flow-app` / `flow.schemati.test` behind the
  dev proxy). Its `/api` proxy targets Laravel, which serves flows, modules, schematics,
  and session auth. Script execution runs in-browser (workers + WASM).
- **`flow-server`** — this repo's Bun API server for **headless execution**
  (`POST /api/execute` with the flow-folding cache). In-network it reaches Laravel at
  `http://schemati-app:80`; on the host it's published at `FLOW_API_PORT` (default 3002).

The `deploy` script builds the client and copies it into schemati's public dir
(only meaningful from the submodule position):

```bash
bun run deploy       # turbo build && cp -r client/dist/ ../public/flow/
```

### Environment variables

| Variable | Where | Purpose |
|---|---|---|
| `VITE_PORT` / `VITE_HMR_HOST` / `VITE_HMR_PROTOCOL` | client | Dev-server port + HMR endpoint (coupled: `flow.schemati.test` / `wss`) |
| `VITE_API_PROXY_TARGET` | client | Where `/api` is proxied. Standalone: `http://localhost:3001` (this repo's server). Coupled: `http://schemati-app:80` (Laravel) |
| `VITE_SCHEMATI_URL` | client | Platform origin for auth/profile links |
| `SCHEMATI_URL` | server | Platform base URL for the `Schemati` ambient + the `/api/v1/schematics` passthrough |
| `SCHEMATI_API_TOKEN` | server | Player JWT (with `upload_schematic` for uploads); `sub` becomes the author |
| `SCHEMATI_TLS_INSECURE=1` | server | Accept self-signed certs (dev `schemati.test`) |
| `FLOW_API_PORT` / `FLOW_SCHEMATI_API_TOKEN` | schemati compose | Host port + token for the `flow-server` service |

### The `Schemati` ambient

Blocks talk to the platform through the `Schemati` runtime provider — environment
branching lives in the provider, not in block code:

```ts
const hits = await Schemati.searchSchematics({ tag: 'door', limit: 10 });
const s = await Schemati.getSchematic(hits[0].shortId);     // live Schematic
await Schemati.uploadSchematic(s, { name: 'Copy', tags: ['door'] });
```

In the browser these calls ride the page's same-origin session (coupled mode); a
session-scoped player JWT is minted automatically for uploads. On the server they
use `SCHEMATI_URL` + `SCHEMATI_API_TOKEN`. Upload previews are generated by a
dependency-free top-down PNG renderer, so no environment needs a canvas.
